# СЦЕНАРИО ЗА ВИДЕО ПРЕЗЕНТАЦИЈА (3 минути)

**Тема:** Информациски систем за регистрација и менаџирање со безбедносни инциденти во SOC
**Автори:** Кристијан Пешевски (226142), Филип Михајлов (214008)
**Предмет:** Менаџмент информациски системи (МИС), ФИНКИ — 2026
**Архитектура:** React ➝ Spring Boot (XML-RPC мост) ➝ Odoo 17 Helpdesk

---

## 🎬 ДЕЛ 1 — React фронтенд: Пријавување на инцидент (0:00 – 1:00)

**[Кадар 1] — Почетен екран на React порталот.**

> **Глас (Кристијан):** „Добредојдовте во нашиот SOC Портал. Ова е веб-апликација изработена во React 19 преку Vite, со axios за комуникација со Spring Boot мидлверот."

**[Кадар 2] — Отворање на формуларот „Пријави Безбедносен Инцидент".**

> **Глас (Кристијан):** „Корисникот пополнува три задолжителни полиња: краток наслов, детален опис и го избира засегнатото ИТ средство од паѓачкото мени. Тука може да се види дека листата доаѓа од Spring Boot преку GET /api/assets."

**[Кадар 3] — Пополнување на формуларот.**

> **Глас (Кристијан):** „Ќе пријавиме сомнителен phishing мејл кој пристигнал на е-пошта на вработен, а засегнатото средство е Главната База на Податоци. Како тип избираме PHISHING, како итност VISOKA."

**[Кадар 4] — Клик на „Поднеси".**

> **Глас (Кристијан):** „Кликаме Поднеси. axios испраќа POST /api/incidents со JSON payload што ги содржи насловот, описот, assetId и типот. По неколку секунди добиваме зелена потврда со Odoo Ticket бројот, на пример INC-2026-0007."

---

## 🎬 ДЕЛ 2 — Spring Boot ➝ Odoo: Автоматска регистрација (1:00 – 2:00)

**[Кадар 5] — Таб на прелистувач: Odoo Helpdesk Kanban (http://localhost:8069).**

> **Глас (Филип):** „Префрлуваме се на Odoo ERP Helpdesk модулот. Како што гледате, новиот тикет е веќе присутен во колоната НОВ. Тоа значи дека Spring Boot мидлверот преку apache-xmlrpc клиент успешно го повика execute_kw('helpdesk.ticket', 'create', ...) на Odoo."

**[Кадар 6] — Отворање на тикетот во Odoo формата.**

> **Глас (Филип):** „Кога ќе го отвориме тикетот, гледаме дека полето Priority автоматски е поставено на 3 (Very High). Тоа е направено од Odoo Studio Automated Action која го пресмета приоритетот врз основа на полето x_criticality на средството — бидејќи Главната База на Податоци е KRITICNA, Odoo автоматски го подигна приоритетот."

**[Кадар 7] — Конзола на Spring Boot (логови).**

> **Глас (Филип):** „Во логовите на Spring Boot гледаме Created Odoo helpdesk.ticket id=7 title='Phishing email...'. Целосниот тек на комуникацијата е: React ➝ HTTP/JSON ➝ Spring Boot ➝ XML-RPC ➝ Odoo."

## 🎬 ДЕЛ 3 — Odoo: Истрага, затворање, SLA (2:00 – 3:00)

**[Кадар 8] — Аналитичарот го презема тикетот.**

> **Глас (Филип):** „Како дежурен аналитичар го преземам тикетот. Odoo автоматски го префрла во фазата Под истрага и го заклучува за преземање од друг аналитичар. SLA тајмерот започна да мери."

**[Кадар 9] — Затворање на тикетот преку Odoo Closure формата.**

> **Глас (Филип):** „Откако ја извршивме санацијата, го отвораме формуларот за затворање. Тука пополнуваме дијагноза — True Positive — и опис на преземените чекори: блокиран испраќач, ресетирана лозинка, едуциран корисник."

**[Кадар 10] — Odoo Dashboard со SLA KPI.**

> **Глас (Кристијан):** „Откако ќе кликнеме Затвори, Odoo преку Studio Automation внесува account.analytic.line (timesheet), пресметува reaction_time и resolution_time, и го логира исполнувањето на SLA. На крајот, тикетот се појавува во колоната РЕШЕН со зелена ознака."

**[Кадар 11] — Odoo Reports / Dashboard.**

> **Глас (Кристијан):** „На dashboard-от сега може да се видат сите KPIs: просечно време на реакција, процент на исполнетост на SLA, најчести типови на инциденти. Ова се податоци кои претходно беа невозможни бидејќи инцидентите се пријавуваа преку е-пошта."
